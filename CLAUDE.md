## What This App Is

A desktop Point of Sale (POS) application for a mobile phone retail shop.
Built with Tauri + React + TypeScript. Data is stored locally in SQLite via
Drizzle ORM. There is no backend server in the MVP вЂ” everything runs on the
client machine, offline-first.

Two types of products are sold:
- **Mobiles** вЂ” tracked individually by IMEI number
- **Accessories** вЂ” tracked by quantity (batch/lot)

Every financial transaction automatically posts double-entry accounting entries
to the general ledger. This is non-negotiable and must never be skipped.

## Coding Conventions

### Data Access
- All database access goes through **Tauri commands** (`invoke()`).
- React components never call DB queries directly.
- Query functions in `src/db/queries/` are called only from Tauri command
  handlers or from within other query functions (e.g. accounting engine).

### Accounting
- Every save of a purchase invoice, sales invoice, or return MUST call
  `postJournalEntry()` from `src/modules/accounting/engine.ts`.
- The accounting engine is a pure function вЂ” it takes a transaction description
  and writes the appropriate `journal_entries` + `journal_entry_lines` rows.
- Never hardcode account names in business logic. Always look up account IDs
  from the `accounts` table using the account `code`.

### TypeScript
- Define types in `types.ts` per module. Shared types go in `src/types/index.ts`.
- Drizzle inferred types (via `typeof table.$inferSelect`) are the canonical
  types for DB rows. Do not redefine them manually.
- Use `z` (Zod) for form validation schemas. Keep them co-located with the
  form component.

### Naming
- Database tables: `snake_case` (Drizzle convention)
- TypeScript types/interfaces: `PascalCase`
- Functions and variables: `camelCase`
- Tauri commands: `snake_case` (Rust convention on the backend, mapped to
  camelCase when called via `invoke()` on the frontend)

### State
- Zustand stores hold **UI state** (form state, loading flags, selected rows).
- Zustand stores do NOT cache DB data. Data is always fetched fresh via Tauri
  commands and stored in local component state or React Query if needed.

## Key Business Rules

These are domain rules that must be enforced at the query/command level,
not just in the UI.

1. **IMEI uniqueness** вЂ” An IMEI cannot appear in two purchase invoices unless
   the first one was fully returned. Enforce this in the purchase save command.

2. **Mobile sales require IMEI** вЂ” When selling an item of `item_type = mobile`,
   an IMEI must be selected. The IMEI must currently have `status = in_stock`.

3. **Accessory sales use quantity** вЂ” No IMEI involved. Deduct from `stock.quantity`.
   Never allow stock to go below zero.

4. **Cost price snapshot** вЂ” When saving a sales invoice line, the current
   purchase cost of the item must be saved into `sales_invoice_lines.cost_price`.
   This value must never be recalculated retroactively. It is the cost at the
   moment of sale.

5. **Accounting entries are mandatory** вЂ” No purchase or sale can be saved
   without a corresponding `journal_entries` record and at least two
   `journal_entry_lines` records (debit + credit). If the accounting write
   fails, the entire transaction must roll back.

6. **Partial payments** вЂ” When `payment_type = partial` on a purchase invoice,
   both a cash entry and a payable entry are created. The partial amounts must
   be captured explicitly (not inferred).

7. **Returns reference originals** вЂ” A purchase return must reference a valid
   `purchase_invoices.id`. A sales return must reference a valid
   `sales_invoices.id`. Standalone returns are not allowed.

8. **No hard deletes** вЂ” Never delete purchase or sales invoices. Use `status`
   fields to mark records as void/returned. Financial records are permanent.

## Implementation Order

Follow this sequence. Do not skip ahead.

1. Drizzle setup + `src/db/schema.ts` вЂ” full schema, all tables
2. Drizzle Kit migrations вЂ” generate and run
3. Seed default Chart of Accounts (see `docs/schema.md` for the default accounts)
4. Accounting engine (`engine.ts`) вЂ” `postJournalEntry()` function
5. Purchase module вЂ” commands, queries, then UI
6. Sales module вЂ” commands, queries, then UI
7. Inventory views вЂ” read-only views driven by purchase/sales data
8. Reports вЂ” P&L, balance sheet, ledger views (post-MVP)
